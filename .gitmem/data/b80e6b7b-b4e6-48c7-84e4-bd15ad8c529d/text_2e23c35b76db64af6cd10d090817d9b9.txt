# рҹҢ… BlackoutOps вҖ” the morning-after incident brain

> **You got paged at 3AM. You fixed it half-asleep. BlackoutOps remembers what you can't.**
>
> Built for **"The Hangover Part AI: Where's My Context?"** вҖ” WeMakeDevs Г— Cognee hackathon, July 2026.
> Runs end-to-end on **Cognee Cloud**. The app keeps **no database of its own** вҖ” Cognee *is* the memory.

![BlackoutOps war room](docs/screenshot-warroom.png)

## The problem

Every on-call engineer knows the hangover: alerts at 02:47, pages at 03:15, a blur of `kubectl` commands, a rollback at 03:41, sleep at 04:02. The next morning the dashboard is green and the *context is gone* вҖ” what broke, what you ran, why it worked, and whether the team has seen this exact failure before. That knowledge normally dies in scrollback. Six months of incidents later, the org keeps re-diagnosing the same eviction storm from scratch.

**BlackoutOps is The Hangover, played straight for DevOps:** it reconstructs the night you can't remember, answers questions about it, learns the confirmed root-cause pattern, and honors retention policy by provably forgetting.

Most incident tooling helps you fight the *next* fire. BlackoutOps rebuilds the *last* one you can't remember вҖ” forensic reconstruction of a lived-through outage is the literal answer to "Where's My Context?", and it's what the whole design is bent toward.

## What it does

1. **Ingests the night's raw artifacts** вҖ” Prometheus alerts, PagerDuty pages, the Slack `#incidents` scroll, your actual shell history, deploy logs вҖ” into a Cognee knowledge graph that connects *services вҶ’ deploys вҶ’ commands вҶ’ causes вҶ’ people*.
2. **Morning-after debrief chat** вҖ” "What happened last night?", "What exactly did I run at 3AM?" вҖ” every answer is a live, auto-routed `recall()` over graph + session memory.
3. **Cross-incident memory** вҖ” "Have we seen this before?" traverses a *separate* incident from three weeks earlier and connects both through shared entities (`redis-cache`, TTL-less keys, eviction storm). This is the question a plain vector store fumbles and a knowledge graph nails.
4. **Files the postmortem back into memory** and runs `improve()` вҖ” the confirmed pattern is enriched so the *next* 3AM page gets answered in seconds.
5. **Forgets on command** вҖ” retention policy as a first-class feature: `forget(dataset=вҖҰ)` surgically deletes an incident, and you can watch recall honestly not know it anymore.

## How we use Cognee (the whole lifecycle, not just RAG)

| Lifecycle | Where | What it does here |
|---|---|---|
| **`remember()`** | [`memory.py вҶ’ ingest_artifacts()`](memory.py) | Each raw ops artifact becomes permanent graph memory in a per-incident dataset (`incident_2026_07_05`, `incident_2026_06_14`) |
| **`remember(QAEntry, session_id=вҖҰ)`** | [`memory.py вҶ’ store_qa()`](memory.py) | Every debrief Q&A is stored as a **typed QAEntry** in session memory (`morning-after-debrief`) вҖ” refresh the page: the *UI* forgets, Cognee doesn't |
| **`remember(FeedbackEntry, вҖҰ)`** | [`memory.py вҶ’ add_feedback()`](memory.py) | рҹ‘Қ/рҹ‘Һ on any answer stores a typed FeedbackEntry attached to that QA memory вҖ” human signal living next to the answer it judges |
| **`recall()`** | [`memory.py вҶ’ ask()`](memory.py) | Auto-routed recall over **both** incident datasets + session memory (`auto_route=True`), returning graph-completion answers with provenance |
| **`improve()`** | [`memory.py вҶ’ file_postmortem()`](memory.py) | The confirmed postmortem goes through the full cloud remember pipeline, then the dedicated `improve()` pass is attempted вҖ” with the honest receipt shown in the UI on tenants that don't expose that route yet |
| **`forget()`** | [`memory.py вҶ’ purge()`](memory.py) | Dataset-scoped deletion (`memory_only=True`) with Cognee's own receipt shown in the UI вҖ” see "Bugs we found upstream" for why memory-only |
| **Graph visualization** | [`memory.py вҶ’ graph_html()`](memory.py) | The tenant's `/api/v1/visualize` rendered in-app, so you can *see* what the brain built out of the night |

- **`memory.py`** вҖ” the honest core: five functions, each mapping 1:1 to a lifecycle call. No local store, no cache, no cheating.
- **`app.py`** вҖ” thin FastAPI: seed / ask / postmortem / forget / graph endpoints + static UI.
- **`dns_fallback.py`** вҖ” real-world ops touch: freshly provisioned tenant subdomains can sit in public resolvers' negative DNS cache (ours did, on 1.1.1.1). If the system resolver fails, we resolve over DNS-over-HTTPS and pin via a `socket.getaddrinfo` patch. Diagnosed and worked around *during* the hackathon вҖ” very much in the spirit of the theme.

## Quickstart

```bash
git clone https://github.com/devanshug2307/blackoutops && cd blackoutops
python3.12 -m venv .venv && .venv/bin/pip install -r requirements.txt

cp .env.example .env   # then fill in your tenant URL + API key

.venv/bin/uvicorn app:app --port 8787 --loop asyncio
# open http://127.0.0.1:8787
# (--loop asyncio lets the DoH DNS fallback work for brand-new tenants;
#  uvloop bypasses socket.getaddrinfo, plain asyncio doesn't)
```

Get a tenant + key at [platform.cognee.ai](https://platform.cognee.ai) (hackathon credit code `COGNEE-35`).
